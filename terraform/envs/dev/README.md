# OPIK Dev Terraform Environment

This environment creates the initial AWS split described in `../../guid.md`.

## Resources

- VPC with public, private app, and private DB subnets across two AZs
- Internet Gateway and NAT Gateway
- S3 Gateway VPC Endpoint
- Public front proxy EC2 for the FastAPI/RAG server when ALB permissions are unavailable
- Private EC2 for Airflow, embedding, Spark/Delta, and batch jobs
- Private EC2 for FastAPI, FAISS, RAG, LangChain/LangGraph
- Two private RDS PostgreSQL instances
  - `airflow`
  - `opik_service`
- Redis container inside the Airflow EC2 for dev Airflow Celery
- EC2 IAM roles for SSM, S3, Bedrock, CloudWatch Logs, Secrets Manager, and SSM Parameter Store
- Secrets Manager containers for runtime app secrets

## Commands

```bash
cd terraform/envs/dev
cp terraform.tfvars.example terraform.tfvars
terraform init
terraform fmt -recursive
terraform validate
terraform plan -out=tfplan
```

Run `terraform apply tfplan` only after confirming the expected cost and public exposure.

## Access

- API: `front_proxy_url`, forwarded by nginx to private EC2 port `8000`
- Airflow UI: use SSM port forwarding to private EC2 port `8080` by default
- SSH: no inbound rule is created

### Airflow UI Access

Airflow is not exposed through the public front proxy. Keep it private and open a local SSM tunnel when needed.

```bash
cd terraform/envs/dev
terraform output -raw airflow_ui_port_forward_command
```

Run the printed command in one terminal. It has this shape:

```bash
aws ssm start-session \
  --region ap-northeast-2 \
  --target "$(terraform output -raw airflow_instance_id)" \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["8080"],"localPortNumber":["8080"]}'
```

Then open:

```text
http://127.0.0.1:8080
```

The default username is `airflow`. The password is generated on the Airflow EC2 unless explicitly provided through the Airflow runtime secret. Retrieve it through SSM:

```bash
INSTANCE_ID="$(terraform output -raw airflow_instance_id)"
COMMAND_ID="$(
  aws ssm send-command \
    --region ap-northeast-2 \
    --instance-ids "$INSTANCE_ID" \
    --document-name AWS-RunShellScript \
    --parameters 'commands=["sudo cat /opt/opik/airflow-admin.env"]' \
    --query 'Command.CommandId' \
    --output text
)"

aws ssm get-command-invocation \
  --region ap-northeast-2 \
  --command-id "$COMMAND_ID" \
  --instance-id "$INSTANCE_ID" \
  --query 'StandardOutputContent' \
  --output text
```

Operational checks:

```bash
INSTANCE_ID="$(terraform output -raw airflow_instance_id)"
aws ssm send-command \
  --region ap-northeast-2 \
  --instance-ids "$INSTANCE_ID" \
  --document-name AWS-RunShellScript \
  --parameters 'commands=["systemctl status opik-airflow --no-pager","docker ps","curl -sS http://127.0.0.1:8080/health"]'
```

## Runtime Configuration Loading

Terraform does not read the repository `.env` file during AWS deployment.

The current dev environment uses this flow:

1. Terraform variables are loaded from CLI flags, `terraform.tfvars`, environment variables named `TF_VAR_*`, or defaults in `variables.tf`.
2. Non-secret infrastructure values are rendered into Terraform resources and outputs.
   - VPC CIDRs, instance types, feature flags, `repo_url`
   - S3 bucket name
   - RDS endpoints
3. RDS passwords are generated by RDS managed master user passwords.
   - Terraform state contains the secret ARN, not the password value.
   - The Airflow EC2 instance profile can read the RDS managed secret at boot.
4. Terraform creates optional runtime secret containers:
   - `opik-dev/airflow/runtime`
   - `opik-dev/api/runtime`
   - Terraform creates the containers only. Runtime secret values are not written by Terraform.
5. Airflow EC2 bootstrapping creates runtime files on the instance:
   - `/opt/opik/bootstrap.env`: non-secret endpoints and ARNs
   - `/opt/opik/airflow.env`: container environment, DB URLs, Airflow keys, DART/Bedrock values
   - `/opt/opik/airflow-admin.env`: Airflow UI credential reference
6. Airflow values are sourced as follows:
   - RDS username/password: read from AWS Secrets Manager RDS managed secrets at boot
   - Airflow Fernet/webserver/admin password: read from `opik-dev/airflow/runtime` if present, otherwise generated on the EC2
   - DART/Bedrock optional values: read from `opik-dev/airflow/runtime` if present, otherwise left empty/defaulted
   - S3/AWS region/endpoints: rendered from Terraform values
7. API EC2 currently writes `/opt/opik/bootstrap.env` and runs `opik-server.service` from that file.
   - It receives S3 bucket, service DB endpoint, runtime secret ARN, FAISS paths, and static app paths.
   - It does not currently expand a full app `.env` from the runtime secret.

Do not place secrets in:

- `terraform.tfvars`
- Terraform variable defaults
- user-data templates as literals
- committed `.env`
- Terraform outputs
- shell command history

## Local Secret Handling Without Cloud Secret Storage

If cloud secret storage is not allowed, do not use a plaintext `.env` as the source of truth.

Use one of these local-first patterns:

1. Recommended: `sops` + `age`
   - Store encrypted config such as `env/dev.enc.yaml` or `.env.enc` in git.
   - Keep the age private key only on trusted operator machines or offline storage.
   - During deployment, decrypt locally and stream the result to the instance as a root-only file.
   - Commit only encrypted files.
2. Simple local option: `gpg` encrypted `.env.gpg`
   - Keep `.env.gpg` in a private backup or private repo.
   - Decrypt locally only during deployment.
   - Write to the EC2 as `0600`, owned by root.
3. Team handoff option: `ansible-vault`
   - Store encrypted variables in the repo.
   - Operators need the vault password out of band.
   - Ansible writes the final env file over an encrypted control channel.

Practical local-only operating pattern:

```bash
sops -d env/dev.airflow.env.enc > /tmp/opik-airflow.env
chmod 600 /tmp/opik-airflow.env

# Transfer through a private admin channel such as VPN/SSH or Ansible.
sudo install -o root -g root -m 600 /tmp/opik-airflow.env /opt/opik/airflow.env
sudo systemctl restart opik-airflow

rm -f /tmp/opik-airflow.env
```

The current dev Terraform intentionally disables inbound SSH and relies on SSM. SSM is safe for access control, but `aws ssm send-command` parameters and command output can be retained in AWS operational history. Do not pass full `.env` contents directly through `send-command` for sensitive production secrets. If cloud secret storage is prohibited, add a private admin connectivity path and use an encrypted local source such as `sops` or `ansible-vault`.

For production, keep `AIRFLOW__CORE__FERNET_KEY` stable. If it is regenerated on every instance replacement, previously encrypted Airflow Variables and Connections can become unreadable.

## Secret Handling

RDS master passwords are managed by AWS Secrets Manager through RDS managed master user passwords.
Application runtime secrets are not written by Terraform. Store values such as Airflow Fernet key,
Airflow webserver secret key, Telegram token, DART keys, and optional Bedrock API token in the
created Secrets Manager containers or SSM Parameter Store after apply.

Do not put secret values in `terraform.tfvars`.
