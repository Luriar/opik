# OPIK 프로젝트 — 개발 패턴 회고

## 평소 스타일 vs 이번 스타일

보통은 "설계 → 구현 → 검증" 순서로 간다. 전체 아키텍처 다이어그램 그리고, 스키마 정의하고, 인터페이스 협의하고, 그다음 코드를 짠다.

OPIK은 달랐다. 시간이 빠듯한 프로젝트에서 "일단 돌아가는 걸 먼저 만들고, 거기서 설계가 나온다"는 패턴으로 진행됐다. 결과적으로 이 방식이 잘 맞았다. 그 과정을 정리한다.

## 0. 전제 조건 — 이 프로젝트가 가진 제약

- 데이터 소스의 성격을 미리 완전히 알 수 없었다. 네이버 증권사 리포트가 어떤 포맷인지, DART API 응답이 어떤 형태인지, 실제 텍스트 추출률이 어느 정도인지는 해봐야 안다.
- 팀원들이 동시에 다른 파트(주가 예측, 공시 수집)를 진행 중이었다. 나 혼자 모든 걸 결정할 수 없고, 인터페이스는 서로의 결과물이 나온 뒤에야 정의할 수 있었다.
- 시간 제약이 있었다. "완벽한 설계"를 기다릴 여유가 없었다.

## 1. 패턴: Build → Validate → Refine → Document → Design Next

### Step 1 — Build: 일단 만든다

설계는 최소한으로, 동작하는 코드를 먼저 만든다.

- 네이버 수집기: API 호출해서 PDF 다운로드. 증권사별 CDN 패턴, EUC-KR 인코딩 같은 건 만들면서 발견했다.
- Silver 추출: PyMuPDF로 PDF → 텍스트. 99.99% 추출 성공이라는 건 67,000건 돌려보고 나서야 알 수 있었다.
- Gold Structured: 정규식으로 투자의견/목표주가/종목코드 추출. 75% TP 추출률이라는 것도 다 돌려보고 나온 숫자다.

이 단계에서 중요한 건 "완성도"가 아니라 "실제 데이터와의 접촉"이다. 데이터를 만져보지 않고는 진짜 문제가 뭔지 모른다.

### Step 2 — Validate: 실제 데이터로 검증한다

만든 코드에 실제 데이터를 태운다. 여기서 진짜 스펙이 드러난다.

- 51,294건을 돌리자 TP false positive가 8.3%였다. PER, 시가총액, 연도, 괴리율 등 온갖 숫자가 목표주가로 잘못 잡혔다.
- 2020년 한국투자증권 데이터가 없다는 걸 발견했다 (네이버에서 PDF 첨부가 안 되는 증권사).
- `발행일` 포맷이 `2026-06-11`과 `2026.06.11` 두 개로 섞여 있다는 걸 이때 알았다.

### Step 3 — Refine: 문제를 발견한 순간 고친다

Validation에서 나온 문제를 하나씩 해결한다. 이 과정에서 설계의 핵심 결정들이 자연스럽게 나온다.

TP false positive 하나를 잡으려고 시작한 게 9-layer defense가 됐다:
```
1. "12개월 목표주가"의 12 → 컨텍스트에서 년/개월/배 키워드 체크
2. "CB 100% 전환" → 뒤에 % 있으면 기각
3. "TP(12M)" → 뒤에 M 있으면 기각
8. "105\n2,000" → 멀티라인 split 감지 (실제 TP는 105,000원)
9. "170,\n700" → 콤마+개행 split 감지 (실제 TP는 170,000원)
```

이 방어 레이어들은 설계 문서에서 나온 게 아니다. 삼성전자 TP=105, 티씨케이 TP=170 같은 실제 엣지케이스를 만나면서 하나씩 쌓아올린 거다. 결과적으로 TP false positive가 8.3% → 0.044%로 떨어졌고, 이 9개 레이어가 Gold Structured의 핵심 설계 자산이 됐다.

### Step 4 — Document: 코드가 안정되면 문서를 쓴다

돌아가는 코드와 검증된 숫자가 있으면, 그걸 바탕으로 문서를 쓴다.

- `HOW_BRONZE_TO_SILVER_WORKS.md` — Bronze→Silver 파이프라인의 구조, 스키마, 실행 명령어, 성능
- `HOW_SILVER_TO_GOLD_WORKS.md` — 추출 로직 5종, 9-layer TP defense, 결측치 처리 원칙, LLM Gold 로드맵

이 문서들은 "앞으로 이렇게 하자"가 아니라 "지금 이렇게 되어 있다"를 기록한 거다. 문서가 구현을 선행하지 않았다. 구현이 문서를 만들었다. 이 순서가 중요하다 — 시간이 없을 때는 검증된 사실만 문서화하는 게 빠르다.

### Step 5 — Design Next: 만들어진 것 위에서 다음을 설계한다

Phase 1이 완성되고 나서야 Phase 2 설계를 시작했다.

핵심은 Phase 1에서 얻은 **실측 데이터**를 설계의 입력으로 쓴다는 거다:
- 일평균 22건, 피크 48건 → 설계 기준 50건/일
- Haiku API 50건 × 20 workers → 5~10초 내 처리 가능
- 51,294건 Parquet 전체가 50MB → Spark 굳이 필요 없음 → 그래도 팀 협업과 Delta Lake 때문에 Spark 도입

EC2 스펙, Airflow DAG 구조, 비용 추정 모두 실제 돌려본 숫자에서 나왔다. "짐작"이 아니라 "측정" 위에 지은 설계다.

## 2. 이 패턴이 잘 맞았던 이유

### 설계가 뒤로 간 게 아니라, 설계의 입력이 달라졌다

보통의 설계는 요구사항을 입력으로 받는다. 이번 프로젝트의 설계는 **실측 데이터**를 입력으로 받았다. "목표주가 추출률은 몇 %일까?" 같은 건 설계 단계에서 추정할 수 없다. 돌려보고 나서야 75.0%라는 숫자가 나오고, 그걸 보고 LLM Gold로 보완할지 말지 결정할 수 있다.

### 빠른 피드백 루프

수집 → Silver → Gold → 챗봇까지의 엔드투엔드를 하루 만에 검증했다(6월 12일). 파이프라인이 실제로 돌아가는 걸 확인한 후에야 "이제 이걸 매일 자동으로 돌리자"는 Phase 2 계획이 의미를 갖는다.

### 문서가 구현을 따라온다

시간이 없을 때 흔히 하는 실수는 "일단 설계부터 완벽하게 하자"다. 근데 데이터를 모르는 상태에서 쓴 설계는 결국 다시 쓰게 된다. 구현 → 검증 → 문서화 순서로 가면 두 번 쓰는 일이 없다.

## 3. 한계와 주의점

이 패턴은 만능이 아니다. 몇 가지 조건에서만 잘 통한다:

- **데이터의 성격을 미리 알 수 없을 때.** 데이터 소스의 구조, 품질, 엣지케이스를 모르는 상태에서 설계하는 건 낭비다.
- **빠른 실험이 가능할 때.** 51,294건을 3분 만에 돌려볼 수 있는 환경이 갖춰져 있었다. 실험이 느리면 이 방식은 답답해진다.
- **혼자 혹은 작은 팀일 때.** 다른 사람과 인터페이스 합의가 필요한 부분은 어쩔 수 없이 설계가 선행해야 한다.

반대로 이런 상황에서는 전통적인 설계-우선이 더 맞다:
- 여러 팀이 동시에 개발해야 하는 큰 프로젝트
- 인프라 구축에 시간이 오래 걸리는 경우 (EC2 프로비저닝, DB 스키마 마이그레이션 등)
- 규제나 보안 요구사항으로 인해 사전 승인이 필요한 경우

## 4. 타임라인

```
Phase 1 — "만들고 부딪히기"
  1일차: 네이버 수집기 + 한국투자증권 수집기 → Bronze 백필 시작
  2일차: Silver 추출 (PyMuPDF) → 51,294건 완료
  3일차: Gold Structured (정규식) → TP false positive 방어 9-layer 완성
  4일차: 엔드투엔드 검증 (6/12 데이터: 수집→Silver→Gold→챗봇)
  5일차: 문서화 (HOW_BRONZE_TO_SILVER_WORKS, HOW_SILVER_TO_GOLD_WORKS)
  6일차: 텔레그램 연동 완료 + Phase 2 설계 시작

Phase 2 — "실측 위에 설계하기"
  7일차: Phase 2 전체 설계 (EC2/Airflow/Spark/LLM/스코어링/텔레그램)
  8일차: DART 공시 파이프라인 설계 (상용님 파트 연동)
```

## 5. 결론

이번 프로젝트의 개발 패턴을 한 문장으로 정리하면:

> **"일단 데이터를 만지고, 거기서 배운 걸로 설계한다."**

설계 → 구현이 아니라 Build → Validate → Refine → Document → Design Next다. 시간이 없고, 데이터를 모르고, 혼자 혹은 작은 팀에서 빠르게 검증 가능한 환경이라면 이 순서가 더 빠르고 더 정확한 결과를 낸다. 이 경험은 앞으로도 비슷한 조건의 프로젝트에서 의식적으로 적용할 가치가 있다.
